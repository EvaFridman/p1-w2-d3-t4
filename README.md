# Задача 4 — сокращатель ссылок

Крепкая сборка дня: сервис коротких ссылок. Пользователь отдаёт длинный URL — получает короткий код; заход по короткому адресу перекидывает на исходную ссылку и считает переходы. Здесь впервые в деле `res.redirect`, router-level middleware и проверяющее middleware через `next()`. Хранение — на диске через `fs.promises`.

## Что за сервис

Каждая короткая ссылка — это запись `{ code, url, visits, createdAt }`: короткий код, исходный адрес, счётчик переходов, время создания. Записи лежат в `links.json`, работа с файлом — через `fs.promises` в отдельном модуле хранения (`storage.js`); маршруты про `fs` не знают. Путь к файлу — через `path.join(__dirname, ...)`.

Приложение делит маршруты на две группы: API для управления ссылками (под префиксом `/api/links`, через `express.Router()`) и один «боевой» маршрут редиректа в корне (`/:code`).

## Маршруты API (роутер под /api/links)

- `POST /api/links` — принять `{ url }`, сгенерировать короткий код, сохранить запись с `visits: 0`, вернуть 201 и созданную запись (с кодом);
- `GET /api/links` — список всех ссылок со статистикой; поддержи сортировку по переходам через query: `GET /api/links?sort=visits` — от самых популярных к непопулярным;
- `GET /api/links/:code` — информация об одной ссылке по коду (url, число переходов), 200; нет — 404. Этот маршрут **не** редиректит, только отдаёт данные;
- `DELETE /api/links/:code` — удалить ссылку, 204.

## Маршрут редиректа (в корне)

- `GET /:code` — найти ссылку по коду. Нашлась — увеличить `visits` на 1, сохранить и сделать `res.redirect(302, <исходный url>)`. Не нашлась — 404.

Используй именно 302 (временный редирект), а не 301: браузер кеширует 301 и перестаёт ходить на твой сервер — тогда переходы не посчитаются. Это прямое следствие семантики статусов из позавчерашней лекции.

## Middleware

- **Router-level логгер.** Через `router.use(...)` повесь на роутер `/api/links` middleware, которое печатает в терминал метод и путь каждого запроса к API. Оно должно срабатывать только на маршрутах этого роутера, а не на редирект в корне — так на практике видно, что middleware можно вешать не только на всё приложение, но и на конкретный роутер.
- **Проверка URL.** На `POST /api/links` повесь middleware `validateUrl(req, res, next)`: если `req.body.url` пустой или не начинается с `http://` либо `https://` — ответить 400 и `next()` не звать (запрос до создания не дойдёт). Если всё хорошо — `next()`.

## Генерация кода

Короткий код — несколько случайных символов из набора `[a-z0-9]` (например 5 символов). После генерации проверь, что такого кода ещё нет среди сохранённых; если вдруг совпал — перегенерируй. `crypto.randomUUID` здесь не подходит — он слишком длинный, ссылка не будет короткой.

## Входные данные

Стартовое содержимое `links.json` (создай файл сама):

```js
[
  { "code": "abc12", "url": "https://nodejs.org/docs", "visits": 4, "createdAt": "2026-07-17T09:00:00.000Z" },
  { "code": "xy789", "url": "https://expressjs.com", "visits": 11, "createdAt": "2026-07-17T09:05:00.000Z" },
  { "code": "qw456", "url": "https://developer.mozilla.org", "visits": 2, "createdAt": "2026-07-17T09:10:00.000Z" }
]
```

Тела для проверки:

```js
const okBody  = { url: 'https://github.com' };
const badBody = { url: 'просто текст без схемы' };
```

## Ожидаемый результат

Сервер поднимается на `http://localhost:3000`. `POST /api/links` с телом `okBody` возвращает 201 и новую запись с сгенерированным пятисимвольным кодом и `visits: 0`; тот же запрос с `badBody` возвращает 400, ссылка не создаётся, и в терминале от логгера видно обе попытки. Заход по короткому адресу — `GET /:code` для существующего кода (проверяется через `curl -I http://localhost:3000/xy789`) отдаёт статус 302 и заголовок `Location` с исходным адресом `https://expressjs.com`, а `visits` этого кода в `links.json` увеличивается на 1 при каждом заходе. Несуществующий код по `GET /:code` даёт 404. `GET /api/links` возвращает все ссылки со статистикой; `GET /api/links?sort=visits` ставит `xy789` (11 переходов) первым, `qw456` (2) последним. `GET /api/links/abc12` показывает данные одной ссылки без редиректа. `DELETE /api/links/qw456` возвращает 204 и убирает запись из файла — после перезапуска сервера её нет. Логгер срабатывает на запросы к `/api/links`, но молчит на редирект-переходы в корне, потому что подключён на уровне роутера, а не приложения. Вся файловая работа — в `storage.js`, в маршрутах `fs` нет.
